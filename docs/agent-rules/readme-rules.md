# readme-rules.md

Read this file when the task touches README, installation instructions, distribution notes, or end-user run instructions.

## Purpose

README files should function as thoughtful guided tours of a project.

They are primarily written for:
- future me,
- curious humans who find the repo,
- and potential users who may want to try the app/tool.

They are **not** primarily:
- contributor onboarding systems,
- architecture dumps,
- AI memory storage,
- enterprise documentation,
- or marketing landing pages.

The goal is clarity, honesty, personality, usefulness, and strong visual communication.

---

# Core Philosophy

A good README should feel like:

> “A real developer explaining a real project they actually use.”

Not:
- startup marketing,
- fake open-source professionalism,
- contributor theater,
- or generic AI-generated template output.

The tone should be:
- calm,
- grounded,
- friendly,
- technically literate,
- lightly conversational,
- and visually clean.

---

# Audience Priorities

In order of importance:

1. Future me
2. Curious developers/users browsing GitHub
3. Potential users of packaged releases
4. AI systems (indirectly, through clarity)

README files should not duplicate:
- AGENTS.md
- WHERE_WE_STAND.md
- planning docs
- architectural decision logs
- internal AI instructions

Link to deeper docs where appropriate instead.

---

# Tone & Writing Style

Preferred voice:
- friendly indie developer,
- practical,
- slightly conversational,
- honest,
- mildly opinionated when evidence supports it.

Good:
> “I built this because I got tired of doing X manually.”

Good:
> “This app assumes a fairly Apple-centric workflow.”

Good:
> “Works well for my own use case, though there are definitely rough edges.”

Avoid:
- exaggerated enthusiasm,
- hype,
- buzzwords,
- fake corporate polish,
- startup language,
- “revolutionary/powerful/cutting-edge” phrasing,
- generic AI filler.

Avoid sounding like:
- a VC pitch deck,
- autogenerated OSS boilerplate,
- or a README factory template.

---

# Personal Voice

README files may include:
- workflow preferences,
- design philosophy,
- tradeoffs,
- personal motivations,
- implementation preferences,
- ecosystem assumptions,
- or constraints.

But only when supported by:
- existing docs,
- code structure,
- project behavior,
- explicit author notes,
- or direct user-provided context.

Do **not** invent:
- emotional motivations,
- personal philosophies,
- opinions,
- design values,
- or fictional origin stories.

If information is unclear:
- ask,
- add a TODO note,
- or omit it.

Never hallucinate developer intent.

---

# Visual Style

The default aesthetic should feel:
- clean,
- restrained,
- Mac-native,
- visually calm,
- and intentional.

Prefer:
- large readable screenshots,
- whitespace,
- clean section hierarchy,
- restrained markdown formatting,
- concise prose.

Avoid:
- emoji spam,
- giant badge walls,
- clutter,
- excessive tables,
- giant callout blocks,
- meme formatting,
- noisy layouts.

README files should feel handcrafted, not autogenerated.

---

# Screenshots & Visuals

Screenshots are strongly encouraged whenever possible.

For GUI apps/tools:
- include screenshots early,
- prefer realistic workflows,
- show the app actually doing meaningful work,
- avoid giant walls of screenshots,
- use a few high-quality images instead.

Screenshot standards:
- consistent appearance mode (light/dark),
- clean sample data,
- avoid personal information,
- avoid desktop clutter,
- consistent sizing/window dimensions where practical,
- readable UI scale,
- visually intentional composition.

Animated GIFs/videos are optional, but should only be used when they genuinely improve understanding.

---

# Project Status Block

Most repos should include a lightweight status block near the top.

Examples of status:
- Active Development
- Mostly Complete
- Archived
- Experimental Prototype
- On Hold

Also include expectation-setting language such as:
- built primarily for personal use,
- packaged releases are the intended install method,
- no guarantees/support promises,
- may contain rough edges.

Tone should be calm and human, not defensive or legalistic.

Example:

```md
## Project Status

Actively developed personal project.

Built primarily for my own workflow, but maybe useful to other people with similar needs.

Packaged releases are the intended way to use it.

Still evolving and definitely contains rough edges.
```

---

# AI-Assisted / Vibe-Coded Disclosure

README files should be upfront about AI-assisted development.

Avoid:
- hype,
- defensiveness,
- AI evangelism,
- apologizing for AI usage.

Preferred tone:
- matter-of-fact,
- transparent,
- grounded.

Good examples:

> Like most of my recent projects, this was built with heavy AI assistance using tools like Codex and Claude.

> The ideas, workflows, testing, and direction are mine. The typing speed definitely isn’t.

---

# README Length Philosophy

README files should generally follow a “medium-length” approach.

They should contain:
- enough information to understand and use the project,
- enough visuals to quickly grasp the workflow,
- enough personality to feel human.

But should avoid becoming:
- entire documentation systems,
- giant architecture references,
- contributor manuals,
- or exhaustive technical specifications.

Link to deeper docs when appropriate.

---

# Recommended Section Types

Not every README needs every section.

Use what fits the project naturally.

Common useful sections:
- Short intro
- Project status
- Screenshots
- What it does
- Why it exists
- How it works (lightly)
- Installation / requirements
- Usage basics
- Design choices / tradeoffs
- Limitations
- Future direction
- Related docs
- License

Avoid rigid templates.

Repos should feel recognizably from the same developer without feeling mechanically identical.

---

# Installation Philosophy

README files should prioritize:
- packaged releases,
- practical setup,
- real-world usage.

Assume:
- users are downloading releases,
- not building from source.

macOS projects should acknowledge realities like:
- unsigned app warnings,
- “Open Anyway” workflow,
- permissions requirements,
- Homebrew prerequisites where applicable.

Do not overcomplicate installation instructions.

Developer setup/build instructions are usually unnecessary unless the repo genuinely benefits from outside contributors.

---

# Technical Detail Philosophy

README files may lightly discuss:
- notable implementation decisions,
- interesting technical constraints,
- framework choices,
- rendering/storage/workflow decisions,
- ecosystem assumptions.

But should avoid:
- architecture deep-dives,
- subsystem documentation,
- giant dependency explanations,
- implementation archaeology.

Use an:
> “If you care how this works...”  
style rather than full engineering documentation.

---

# Limitations & Honesty

README files should explicitly acknowledge:
- assumptions,
- workflow constraints,
- ecosystem lock-in,
- rough edges,
- incomplete areas,
- scaling limitations,
- testing limitations,
- maturity level.

Prefer:
> honest constraints

over:
> implied universality.

Good:
> “This assumes a pretty Apple-centric workflow.”

Good:
> “Only really tested on my own machines.”

Good:
> “Optimized for my workflow first.”

This increases trust and clarity.

---

# Roadmaps & Future Plans

Avoid giant roadmap checklists.

Prefer lightweight future-direction sections such as:
- things I still want to improve,
- future ideas,
- planned capabilities,
- current direction.

Archived projects may omit this entirely.

---

# Anti-Patterns

Avoid:
- generic AI tone,
- fake enthusiasm,
- excessive feature bullet lists,
- giant badge collections,
- contributor boilerplate,
- code-of-conduct theater,
- empty “enterprise OSS” sections,
- marketing language,
- fake polish,
- repetitive structure,
- meaningless adjectives,
- aspirational claims,
- hallucinated motivations/philosophy.

If a section adds no real value, remove it.

---

# Final Principle

The README should leave someone feeling:

> “I understand what this project is, why it exists, what state it’s in, and whether I’d enjoy using it.”

Not:

> “I just read a generated software brochure.”
